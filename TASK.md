# Frappe Job Manager — Technical Task

## 1. Идея проекта

Кастомное приложение внутри **Frappe Framework**, которое управляет задачами (`jobs`):

- создание задач;
- запуск задач;
- асинхронное выполнение;
- получение результата;
- базовая аналитика.

Это мини-система обработки фоновых задач, похожая на механики в ERP/CRM-системах.

## 2. Цель проекта

Показать на практике:

- работу с Frappe (`DocType`, `hooks`, API);
- понимание backend-архитектуры;
- асинхронность и очереди;
- интеграции с внешними API;
- работу с БД (MariaDB);
- REST API;
- базовую frontend-логику в Frappe Desk.

## 3. Общая архитектура

### 3.1 Компоненты

```text
[ Frappe Desk UI ]
        ↓
[ Frappe Backend (Python) ]
        ↓
[ Background Worker ]
        ↓
[ MariaDB ]
        ↓
[ External API ]
```

### 3.2 Поток выполнения

1. Пользователь создаёт задачу через UI.
2. Задача сохраняется в БД (DocType).
3. Пользователь нажимает `Run`.
4. Задача отправляется в очередь (`enqueue`).
5. Worker выполняет задачу.
6. Результат сохраняется в БД.
7. UI отображает актуальный статус и результат.

## 4. Основная сущность: `Task` (DocType)

### 4.1 Поля

- `name` (ID от Frappe);
- `task_name` (`Data` / string);
- `task_type` (`Select`);
- `status` (`Select`);
- `payload` (`Text` / JSON);
- `result` (`Text` / JSON);
- `attempts` (`Int`);
- `error_message` (`Text`);
- `created_at`;
- `updated_at`.

### 4.2 Возможные значения

`status`:

- `Pending`
- `Running`
- `Done`
- `Failed`

`task_type`:

- `fetch_weather`
- `fetch_currency`
- `custom`

## 5. Жизненный цикл задачи

1. Создание → `status = Pending`.
2. Запуск → `status = Running`.
3. Выполнение:
   - успех → `Done`;
   - ошибка → `Failed`.
4. Retry (при необходимости).

Retry-политика:

- если `attempts < 3`, допускается повторный запуск.

## 6. Backend-логика

### 6.1 API-слой

Базовый REST от Frappe (автоматически):

- `GET /api/resource/Task`
- `POST /api/resource/Task`

Кастомные методы:

- `POST /api/method/job_manager.run_task`
- `GET /api/method/job_manager.stats`

### 6.2 Сервисный слой

Разделение ответственности:

- `api.py` — входные HTTP-методы;
- `services.py` — бизнес-логика;
- `workers.py` — исполнение фоновых задач.

### 6.3 Асинхронность

Используется встроенный механизм Frappe:

- `frappe.enqueue(...)`

Worker запускается отдельным процессом:

- `bench worker`

### 6.4 Что делает worker

- берёт задачу из очереди;
- меняет статус;
- выполняет задачу;
- сохраняет результат или ошибку.

## 7. Типы задач (интеграции)

1. `Fetch Weather`
   - обращается к погодному API;
   - сохраняет результат.
2. `Fetch Currency`
   - получает курс валют;
   - сохраняет результат.
3. `Custom Task`
   - имитирует полезную нагрузку (`sleep` / mock-операция).

## 8. Обработка ошибок

```python
try:
    run_task()
except Exception:
    status = "Failed"
    error_message = "..."
    attempts += 1
```

Правила:

- все ошибки логируются;
- задача помечается как `Failed` при исключении;
- счётчик `attempts` увеличивается при каждом неуспешном запуске.
